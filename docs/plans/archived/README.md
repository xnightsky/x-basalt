# 归档标准

> 归档 = 从 `docs/plans/` 移入 `docs/plans/archived/`。**不是删除**，文件仍在仓库中、git history 完整。
> 归档的目的是让 `docs/plans/` 保持活跃 plan 的"可见集"，archive 作为历史决策的"可查集"。

---

## 归档准入标准（三问）

一个 plan 满足 **全部三个条件** 才可归档：

### 1. 实现完成

- 所有 Checklist 项目已 ✅
- 对应的代码已落地并通过验证（typecheck / test / build）
- @2026-07-22 为止所有 plan 已经实现完成（社区共识）

### 2. 无唯一信息丢失

归档前必须确认 plan 中的所有**不可从其他真相源还原的内容**已迁移或记录完毕。

需要检查的信息类型：

| 信息类型 | 默认保留在 plan 中 | 可转移到的位置 |
|---|---|---|
| 设计决策与备选方案评估 | 保留 | `docs/specs/<topic>-design.md` |
| 边界收敛声明（有意不做的范围） | 保留 | plan 顶部 `## 范围` 块 |
| 业界调研与信源 | 保留 | `docs/research/` 或 spec 引用 |
| 已知限制与后续拓展方向 | 保留 | plan 尾部 `## 风险与剩余不确定` |
| 阶段切口理由 | 保留 | 如有更好的真相源则转移 |
| 纯执行进度（何时做了什么） | 可归档 | 代码提交历史即可还原 |

**判定原则**：如果该 plan 删除后，某个设计决策或边界声明的"为什么"在仓库中再也找不到了，则不能归档。

### 3. 内容已被替代

- 高层路线图被更细粒度的模块 plan + spec 完全覆盖
- 早期 MVP 设想已被实际实现和迭代后的文档取代
- 同系列 plan 已被合并后的统一文档替代

---

## 不需要归档的情况

以下情况 plan 应**留在 `docs/plans/`**，最多更新 frontmatter 标记 `status: completed`：

- 有**唯一决策记录**（为什么选 A 不选 B、为什么不做 X）
- 有**后续拓展需要的边界信息**（已知限制、未实现的子能力列表）
- 有**可追溯的收敛声明**（"P1 不做 anchor 校验，留 P1.5"）
- 属于**系列 plan 的一部分**（如 KB compiler P0→P3b 五连，整体归档才合理）

---

## 操作规范

```bash
# 归档一个 plan：
git mv docs/plans/<name>.md docs/plans/archived/

# 归档前必须做：
# 1. 对照三问 checklist
# 2. 确认无唯一信息丢失
# 3. 在 commit message 中注明归档理由
```
